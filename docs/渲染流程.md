# 渲染流程

本文说明当前程序入口、DSL Runtime、脏区渲染、DPI 处理和事件等待策略。

## 文件职责

入口在 `main.cpp`。

`main.cpp` 只负责框架级工作：

- 初始化 GLFW / GLAD。
- 创建 OpenGL 3.3 Core 窗口。
- 开启 VSync：`glfwSwapInterval(1)`。
- 监听 framebuffer resize / refresh / content scale。
- 计算 `dpiScale` 和 `pointerScale`。
- 调用 app 生命周期。
- 根据 `app::isAnimating()` 决定 `glfwPollEvents()` 或 `glfwWaitEvents()`。

具体 UI 不写在 `main.cpp`，而是在 `app/*.cpp` 中声明。

## App 生命周期

底层接口在 `app/app.h`：

```cpp
namespace app {

const char* windowTitle();
int initialWindowWidth();
int initialWindowHeight();
bool initialize(GLFWwindow* window);
bool update(GLFWwindow* window,
            float deltaSeconds,
            int windowWidth,
            int windowHeight,
            float dpiScale,
            float pointerScale);
bool isAnimating();
void render(int windowWidth, int windowHeight, float dpiScale);
void shutdown();

}
```

DSL app 通常不直接实现这些函数，而是包含 `app/dsl_app.h` 后只实现：

```cpp
const DslAppConfig& dslAppConfig();
void compose(core::dsl::Ui& ui, const core::dsl::Screen& screen);
```

## 主循环

主循环核心：

```text
计算 deltaSeconds
处理 ESC
读取 framebuffer size
计算 dpiScale / pointerScale
changed = app::update(...)
如果 changed 或窗口要求刷新:
    app::render(...)
    glfwSwapBuffers(...)
如果 app::isAnimating():
    按显示器刷新间隔 sleep
    glfwPollEvents()
否则:
    glfwWaitEvents()
```

目标：

- 空闲时不持续占用 CPU/GPU。
- UI 状态变化时才渲染。
- 动画期间短时间连续刷新。
- 动画刷新节奏尽量贴近显示器刷新率。

## DSL Runtime 更新流程

`app/dsl_app.h` 中的 update 大致流程：

```text
composeFrame()
changed = runtime.update(...)
如果 runtime.needsCompose():
    composeFrame()
    runtime.markFullRedraw()
return changed
```

说明：

- 每帧会先 compose 当前 UI 声明并 layout。
- `Runtime::update()` 读取 pointer，推进 hover / press / transition。
- hover / press / transition 这类纯视觉变化不会触发二次 compose。
- `onClick` 回调通常会修改 app 状态，此时 Runtime 标记 `needsCompose()`，app 再 compose 新 UI，并保守 full redraw。

## 脏区渲染

当前 Runtime 已实现脏区渲染，不再每次交互都全量重绘窗口。

核心策略：

1. Runtime 按 id 缓存每个 `Rect` / `Text` 的动画状态和 primitive。
2. 元素视觉值变化时，计算变化前后的视觉矩形并取 union。
3. 把 union 记录到 dirty rect 列表。
4. Runtime 持有一个离屏 framebuffer cache。
5. full redraw 时把整棵 UI 渲染进 cache。
6. 局部变化时：
   - 绑定 cache framebuffer
   - 对 dirty rect 开启 `glScissor`
   - 清理脏区背景
   - 只渲染与 dirty rect 相交的元素
   - 把 cache blit 到默认 framebuffer

这意味着：

- 写页面和组件时，只要使用 DSL 图元，就自动受益于脏区渲染。
- hover / press / transition 默认只重绘相关区域。
- 页面切换、窗口尺寸变化、点击导致结构变化时会保守 full redraw。

当前脏区是保守实现：复杂重叠和阴影/blur 会扩大重绘区域，以保证视觉正确。

## Full Redraw 触发

这些情况会触发 full redraw：

- 首帧。
- framebuffer 尺寸变化。
- content scale 变化。
- Runtime cache 创建或重建。
- click 回调导致 UI 结构或状态变化后重新 compose。
- 调用 `Runtime::markFullRedraw()`。

## DPI Scale 与 Pointer Scale

当前有两个 scale，不要混用。

`dpiScale` 来自：

```cpp
glfwGetWindowContentScale(window, &scaleX, &scaleY);
```

用途：

- 把 UI 逻辑尺寸转换为 framebuffer 像素。
- 缩放字号。
- 缩放圆角、边框、阴影、blur、transform translate。

`pointerScale` 来自：

```cpp
framebufferSize / windowSize
```

用途：

- 把 GLFW cursor position 转成 framebuffer 坐标。
- 给 hit-test 使用。

两者不一定完全相同。

## 动画与事件等待

`app::isAnimating()` 来自 Runtime：

- hover/press 缓动未结束。
- transition 仍处于 active。

动画中：

```text
sleep 到接近下一个刷新周期
glfwPollEvents()
```

非动画中：

```text
glfwWaitEvents()
```

因此静态界面不会持续刷新。

## 当前限制

- 脏区渲染仍然是保守矩形，不是像素级 diff。
- 没有独立 layer tree，所有内容共用一个 cache framebuffer。
- 没有 clip / scroll，因此脏区不会被父级裁剪。
- 重叠元素为了正确性可能被一起重绘。
- blur 会读取 framebuffer 内容，复杂 blur 区域仍然较贵。
